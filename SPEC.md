# SPEC.md — 중고거래 플랫폼 구현 명세서 (DB Term Project)

> 이 문서는 **데모 시연 평가표(총 300점)** 를 구현 기준으로 정리한 것이다.
> 모든 화면과 기능은 아래 **수락 기준(Acceptance Criteria)** 을 그대로 만족시키는 것을 목표로 한다.
> 각 항목 끝의 `[N점]` 은 데모에서의 배점이며, 곧 우선순위다.

---

## 0. 데모 시나리오 개요

- **등장 회원**: 판매자 A · 구매자 B · 구매자 C · 관리자(회원번호 `c0`)
- **진행 방식**: 한 물품의 거래 흐름을 단계 1 → 9 순서대로 따라가며 시연한다. (1인당 10~20분)
- **중요**: 각 기능을 점검 받을 수 있는 **충분한 데이터(seed)** 를 미리 준비해야 한다.
  최소 — 회원 4명(A, B, C, c0) + 물품 여러 개 + 구매 요청 여러 건 + 채팅 메시지 여러 건.

---

## 1. 데이터 모델 (요약)

**실제 Oracle DB가 유일한 정본(source of truth)** 이다 (CLAUDE.md의 [스키마 파악] 지침대로 DB를 직접 조회). 아래는 의도 파악용 요약이며, 컬럼·타입·제약이 다르면 실제 DB를 따른다.

| 테이블 | 핵심 컬럼 | 메모 |
|---|---|---|
| **Customer** | `cno`(PK), `passwd`, `nickname`(UNIQUE), `phone`, `region` | 관리자는 `cno = 'c0'` |
| **Item** | (`cno`,`itemNo`)(PK), `title`, `description`, `category`, `price`, `tradePlace`, `regDateTime`, `resDateTime`, `sellStatus`, `pic1~3`(BLOB), `finalPrice` | `itemNo` 는 판매자별 1,2,3… 수동 부여 |
| **PurchaseReq** | (`requestCno`,`cno`,`itemNo`)(PK), `reqDateTime`, `reqPrice`, `reqMessage` | 한 물품에 여러 요청자 가능 |
| **ChatRoom** | `roomNo`(PK, IDENTITY), `receiveCno`, `createDateTime`, (`cno`,`itemNo`) | (`receiveCno`,`cno`,`itemNo`) UNIQUE → 방 중복 방지 |
| **Message** | (`roomNo`,`seqNo`)(PK, seqNo IDENTITY), `sender`('S'/'B'), `sentDateTime`, `content`, `isRead`('Y'/'N') | 'S'=판매자, 'B'=구매자 |

**상태 값(고정, CHECK 제약)**
- `Item.sellStatus` ∈ `'판매 중'`, `'예약 중'`, `'거래 완료'`
- `resDateTime` = 예약 승인 시각. **48시간 자동취소 타이머의 기준점.**

## 2. 단계별 수락 기준

### 단계 1 — 로그인 & 상태 구분 (20점)
- [ ] 회원번호 + 비밀번호 입력으로 로그인된다. **[5점]**
- [ ] 로그인 **전/후 화면이 구분**되고, **관리자(c0)** 와 **일반 회원**이 할 수 있는 일이 서로 다르게 처리된다. **[15점]**
  - 관리자: 통계 화면 접근 가능. 일반 회원: 물품 등록/검색/구매요청/채팅.

### 단계 2 — 물품 등록 (35점)
- [ ] 등록 시 **제목·설명·카테고리·가격·거래 희망 장소·등록 일시·판매 상태**가 빠짐없이 DB에 저장된다. **[20점]**
  - 등록 직후 DB에서 저장 여부를 확인할 수 있어야 한다.
- [ ] 한 물품에 **사진 최대 3장**까지 업로드되고, 올린 사진이 화면에 **다시 표시**된다. **[15점]**
  - 사진은 Oracle `BLOB`(pic1~3)에 저장한다.

### 단계 3 — 물품 검색 (50점)
- [ ] **제목 / 카테고리 / 가격 범위** 중 한 가지 조건으로 검색하면 알맞은 물품이 나온다. **[15점]**
- [ ] 여러 조건을 **AND(그리고)·OR(또는)·NOT(제외)** 으로 조합해 검색해도 결과가 정확하다. **[25점]**
  - RISS 상세 검색 형태 참고. WHERE 절을 동적으로 조립한다.
- [ ] 검색 결과를 **최신순·가격순** 등 원하는 기준으로 **정렬**할 수 있다. **[10점]**

### 단계 4 — 구매 요청 (여러 명) (25점)
- [ ] 판매 중인 물품에 **요청 금액 + 메시지**를 담아 구매 요청을 보낼 수 있고, 금액과 메시지가 각각 저장된다. **[15점]**
- [ ] 한 물품에 **여러 회원(B, C)** 이 각각 요청할 수 있고, 판매자 A가 **받은 요청 목록**을 볼 수 있다. **[10점]**

### 단계 5 — 1:1 채팅 (45점)
- [ ] 한 물품에 대해 **B-A 방과 C-A 방이 각각 따로** 생기고, 같은 구매자·판매자 조합의 방은 **중복 생성되지 않는다**. **[10점]**
  - DB의 UNIQUE 제약으로 보장.
- [ ] 각 메시지에 **보낸 사람·보낸 시간·내용·읽음 여부**가 정확히 저장·표시된다. **[15점]**
- [ ] 회원이 **자신이 참여한 채팅방 목록**을 볼 수 있다 (A의 목록에 B 방, C 방이 모두 보임). **[10점]**
- [ ] 각 채팅방마다 **안 읽은 메시지 개수**가 정확히 표시된다. **[10점]**

### 단계 6 — 거래 승인 → 예약 중 (25점)
- [ ] 판매자 A가 B의 요청을 **승인**하면:
  - 물품 상태가 **'예약 중'** 으로 바뀐다.
  - **나머지 회원(C)의 구매 요청은 자동으로 삭제**된다.
  - 승인 시각이 `resDateTime` 에 기록된다(48시간 타이머 시작). **[25점]**

### 단계 7 — 거래 완료 (20점)
- [ ] 거래가 끝나면 물품 상태가 **'거래 완료'** 로 바뀐다. **[10점]**
- [ ] 판매자 A가 **실제 최종 거래 금액(finalPrice)** 을 입력·저장할 수 있다. **[10점]**

### 단계 8 — 예약 자동 취소 (48시간 초과) (25점)
- [ ] 예약(승인)된 지 **48시간이 지나면**, 그 물품이 다시 **'판매 중'** 으로 돌아오고 해당 예약(구매 요청)이 삭제된다. **[25점]**
  - **시연 방법**: 예약 중인 물품 하나의 `resDateTime` 을 SQL 한 줄로 48시간 이전으로 바꾼 뒤 화면 새로고침 → 자동 복귀 확인.
  - 구현 힌트: 화면 진입/새로고침 시 `resDateTime` 이 48시간 초과한 '예약 중' 물품을 찾아 '판매 중'으로 되돌리는 로직을 실행.

### 단계 9 — 관리자 통계 (20점)
- [ ] **그룹 함수**(GROUP BY 합계·평균·건수 등) 통계가 운영에 도움되는 정보를 보여준다. 예: **카테고리별 거래 건수**. **[10점]**
- [ ] **윈도우 함수**(순위·누적합 등) 통계가 의미 있는 분석을 보여준다. 예: **회원별 거래액 순위**. **[10점]**
  - 통계 질의는 본인의 TP-5 산출물이 있으면 그대로 사용하고, 없으면 이 기준에 맞춰 새로 작성한다 (Oracle: `GROUP BY ... ROLLUP`, `RANK() OVER (...)`).

---

## 3. 전반 평가 (데모 전 과정 관찰) (35점)

- [ ] **예외 처리**: 잘못된 입력/허용되지 않는 동작 시, 시스템이 멈추거나 오류 화면을 띄우지 않고 **안내 메시지**로 처리한다. **[15점]**
  - 반드시 처리할 케이스:
    - 필수 항목을 비우고 등록
    - 자기 물품에 구매 요청
    - 이미 거래 완료된 물품에 구매 요청
- [ ] **UI 편의성**: 설명 없이도 쉽게 쓸 수 있다. **[20점]**
  - 목록이 표/카드로 깔끔하게 정리됨
  - 검색·정렬을 버튼으로 선택
  - 판매 상태가 한눈에 보임(상태 배지)

---

## 4. Definition of Done

- 위 단계 1~9 + 전반 평가의 **모든 체크박스**가 데모 시나리오에서 실제로 동작한다.
- 시연 시나리오대로 진행할 수 있는 **seed 데이터**가 준비되어 있다.
- 잘못된 입력에도 앱이 죽지 않고 안내 메시지로 처리된다.
- **배점 합계 검증**: 20+35+50+25+45+25+20+25+20+35 = **300점**.
